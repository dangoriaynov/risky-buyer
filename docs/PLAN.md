# Problem Client — план и спецификация

WordPress/WooCommerce плъгин за маркиране на **проблемни клиенти** (по телефон/име),
с автоматично отбелязване на техните поръчки и архитектура, подготвена за бъдеща
**споделена централна база** между много сайтове.

> Език на интерфейса: български (аудитория на магазина). Документацията — смесена.

---

## 1. Цел

- Поддържаме списък на проблемни клиенти: **телефон и/или име**, с **причина** и **бележка**.
- Поръчките на такива клиенти се **отбелязват автоматично** в списъка с поръчки
  (визуално различно от маркера за дубликати — предупреждение, не „дубликат").
- В бъдеще: всички сайтове с този плъгин се включват в **една обща система** и виждат
  едни и същи проблемни клиенти. **Добавят всички**; **редактират/трият само
  администраторите на ресурса**.
- Засега данните се пазят **локално**, но с правилни абстракции за по-късна централизация.

## 2. Архитектура

```
UI (orders list, order metabox, admin page, AJAX)
        │
        ▼
PC_Blacklist  ← сервиз: нормализация, права, CRUD, matching
        │  използва само интерфейса
        ▼
PC_Storage_Provider (interface)
   ├── PC_Local_Table_Provider   ← сега (таблица wp_pc_blacklist)
   └── PC_Remote_Api_Provider     ← бъдеще (централен сервиз)  [само контракт]
```

- Подмяна на хранилището: филтър `pc_storage_provider`.
- Правата се проверяват в **сервиза** (над провайдера), за да е единна логиката и в
  бъдещия федериран режим.

## 3. Модел на данните — `wp_pc_blacklist`

| Поле | Тип | Бележка |
|------|-----|---------|
| `id` | BIGINT PK | локален |
| `uuid` | CHAR(36) UNIQUE | стабилен ID за крос-сайтова синхронизация |
| `phone_norm` / `phone_raw` | VARCHAR | нормализиран (последни 9 цифри) + оригинал |
| `name_norm` / `name_raw` | VARCHAR | нормализирано + оригинал |
| `reason_code` | VARCHAR | `uncollected` / `fake` / `abusive` / `other` |
| `note` | TEXT | свободна бележка |
| `source_site` | VARCHAR | домейн на сайта, който е добавил записа |
| `status` | VARCHAR | `active` / `removed` (мек delete за история/синхронизация) |
| `created_by` / `created_by_name` / `created_at` | | кой и кога |
| `updated_by` / `updated_at` | | последна редакция |

Причина → цвят: неизкупена 🟠 `#e08a00`, фалшива 🔴 `#d63638`, обиди 🔴 `#9b1c1c`, друго ⚪ `#6b7280`.

## 4. Съвпадение

- Нормализация: телефон = последни 9 цифри; име = малки букви без пунктуация, ≥3 знака.
- Поръчка съвпада, ако **телефон ИЛИ име** е в активния списък (телефонът има предимство).
- В списъка с поръчки: четат се само нужните полета за текущия статус (cap 1500) →
  работи **през пагинацията**.

## 5. UI

- **Списък с поръчки:** ляв цветен бар + бадж `⚠ Проблемен клиент · <причина>`.
- **Екран на поръчка (metabox):** при съвпадение — предупреждение (причина, бележка,
  кой/кога/източник) + бутон „Премахни" (само админ). Иначе — форма „Маркирай клиента".
- **Админ страница „Проблемни клиенти":** таблица + търсене/филтър по причина; форма за
  добавяне (екип); редакция и триене (само админ).

## 6. Права

| Действие | Право |
|----------|-------|
| Добавяне | `edit_shop_orders` (екип) |
| Редакция / триене | `manage_options` (админ) |

В бъдещия федериран режим провайдерът ще връща per-entry флагове `can_edit` / `can_delete`
(чужди сайтове не се редактират), а UI ще ги спазва.

## 7. Бъдеща централизация (само контракт, не се строи сега)

`PC_Remote_Api_Provider` срещу централен сервиз:

- **REST**: `GET /entries` (pull), `POST /entries` (add/push), `PATCH /entries/{uuid}`,
  `DELETE /entries/{uuid}`.
- **Auth**: API ключ/токен на сайт.
- **Source of truth**: централата. Локално — кеш + периодичен pull; добавянето веднага
  push-ва.
- **Конфликти**: по `uuid` + `updated_at`; собственост по `source_site`; редакция/триене
  на чужд запис — само от админ на централата.

## 9. Деплой (текущ процес)

- Кодът се води в това репо (`dangoriaynov/problem-client`), commit/push по SSH ключ.
- Засега качваме промените **ръчно** на сайта (папка `wp-content/plugins/problem-client/`).
- Активирането в WP — по-късно, когато е готово „хубаво".
